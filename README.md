# Лабораторная работа №1: Источники задач и контракты

## Описание
#### Данный проект реализует подсистему приема задач с использованием контрактного программирования:
 * Duck Typing: взаимодействие с объектами на основе их поведения, а не иерархии наследования.
 * Protocol: использование typing.Protocol для определения формальных интерфейсов.
 * Runtime Validation: проверка соблюдения контракта в процессе выполнения программы.
 * Расширяемость: возможность добавления новых источников задач (API, БД, генераторы) без изменения логики обработчика.
## Структура проекта

```
src/
├── processor.py             # Обработчик задач
├── protocol.py              # Протокол
├── sources.py               # Реализации источников задач
└── main.py                  # Точка входа
└── tasks.txt                # Пример файла

tests/                       # Тесты для каждого класса
├── test_api_souce.py        # Обработчик задач
├── test_file_source.py      # Протокол
├── test_generator_source.py # Реализации источников задач
└── tes_processor.py         # Точка входа
```

## Принцип работы
### Программа демонстрирует работу с тремя типами источников, которые не имеют общего предка, но реализуют метод get_tasks():
 * FileSource — считывает список задач из текстового файла.
 * GeneratorSource — программно генерирует заданное количество задач.
 * APISource — имитирует получение данных из внешнего сервиса (заглушка).
Реализованные компоненты
Контракт (src/protocol.py)
 * TaskSource(Protocol) — интерфейс с декоратором @runtime_checkable. Требует наличия метода get_tasks() -> list[str].
Источники задач (src/sources.py)
 * FileTaskSource — работа с файловой системой.
 * GeneratorTaskSource — генерация данных "на лету".
 * ApiTaskSource — эмуляция сетевого взаимодействия.
Обработчик (src/processor.py)
 * TaskProcessor — класс, который принимает любой объект, подходящий под протокол TaskSource.
 * Реализует Runtime-проверку через isinstance(source, TaskSource), гарантируя безопасность выполнения.
Тестирование
Все компоненты покрыты unit-тестами с использованием pytest. Тестирование включает:
 * Проверку корректности возвращаемых данных от каждого источника.
 * Валидацию того, что классы соответствуют протоколу.
 * Тестирование TaskProcessor на устойчивость к некорректным источникам (которые не реализуют контракт).
Памятка пользователю
Для запуска демонстрации используйте main.py.
Доступные источники для проверки:
 * File: чтение из tasks.txt.
 * Generator: создание списка тестовых строк.
 * API: получение статических данных из заглушки.
### Программа выведет список всех собранных задач из разных источников, подтверждая, что TaskProcessor успешно взаимодействует с ними через единый интерфейс.

# Лабораторная работа №2: Модель задачи — дескрипторы и @property

## Описание
#### Проект реализует доменную модель задачи с инкапсуляцией и защитой инвариантов:
 * Пользовательские data-дескрипторы валидируют поля `description`, `priority`, `status` при присваивании.
 * Non-data дескриптор для вычисляемого `is_ready` (разница с data-дескрипторами: присваивание может затенить вычисление и попасть в `__dict__` экземпляра).
 * `@property` для идентификатора и времени создания: `id` задаётся один раз, `created_at` доступен только для чтения.
 * Специализированные исключения (`TaskValidationError`, базовый `TaskError`) при нарушении правил.

## Структура проекта (фрагмент для лаб. №2)

```
src/
├── model.py                 # Класс Task, property для id и created_at
├── descriptors.py           # Data- и non-data дескрипторы с валидацией
├── exceptions.py            # Исключения домена задач
└── main.py                  # Точка входа (лаб. №1 + краткая демонстрация Task)

tests/
└── test_task_model.py       # Тесты модели Task и дескрипторов
```

## Принцип работы
### Атрибуты `Task`
 * `id` — целое число > 0, после создания не меняется (`AttributeError` при повторной установке).
 * `description` — строка длиной от 5 до 200 символов.
 * `priority` — целое от 1 до 5.
 * `status` — одно из: `pending`, `in_progress`, `completed`.
 * `created_at` — время создания (`datetime`), только чтение.
 * `is_ready` — вычисляется: задача не в статусе `completed` и описание валидной длины.

Реализованные компоненты
Модель (`src/model.py`)
 * Класс `Task` с публичным API и внутренним состоянием (`_id`, `_created_at`).
Дескрипторы (`src/descriptors.py`)
 * `BaseDescriptor`, `PriorityDescriptor`, `StatusDescriptor`, `DescriptionDescriptor` — data-дескрипторы с `__get__` / `__set__` и `validate`.
 * `ReadyDescriptor` — non-data дескриптор (только `__get__`), вычисляет готовность к выполнению.
Исключения (`src/exceptions.py`)
 * `TaskError`, `TaskValidationError` для ошибок валидации и домена.

Тестирование
Компоненты лаб. №2 покрыты unit-тестами (pytest): создание и инварианты `id`, валидация полей дескрипторами, поведение `is_ready`, доступ к дескрипторам через класс, иерархия исключений. Целевое покрытие кода пакета `src` при прогоне с `pytest-cov` — не ниже 80%.

Памятка пользователю
Запуск `main.py` дополнительно создаёт пример `Task` и печатает краткую сводку по полям и `is_ready` до и после смены статуса на `completed`.

## Запуск

```bash
python -m src.main
```

Тесты с отчётом о покрытии:

```bash
uv run pytest tests/ --cov=src --cov-report=term-missing
```
